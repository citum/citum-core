## Executive Summary

`citum-engine` is in materially better condition than at the two preceding
audits. The largest architectural concern from those reviews—an implicit,
interior-mutable `Processor` state machine—has been replaced by explicit
`RunState`/`FinalizedRun` typestate. Sorting now uses cached keys, session
reference resolution is cached, render configuration is shared through
`Arc`, and bibliography entry rendering can be compiled with an opt-in
parallel path. These changes improve reuse, make ordering constraints visible
in types, and remove substantial repeated setup work.

No critical finding or unsafe-policy violation was found. Production `unsafe`
remains isolated to the feature-gated C ABI module, whose exported functions
document their caller obligations and validate null pointers and UTF-8 before
use. The normal engine, API, processing, value, sorting, and rendering paths
remain safe Rust.

One high-severity correctness bug was found and fixed during this audit:
citation sorting silently removed missing-reference items, allowing a partial
successful render instead of the documented `ReferenceNotFound` error. Two
medium follow-ups remain: document bibliography output is still generated by
duplicated render passes, and year-suffix ordering does not use the configured
multilingual/locale sort-key policy used by the final bibliography.

## Scope and Method

This is a delta-oriented third review, not a repetition of the two exhaustive
July 3–4 passes. It covers the current crate (48,855 lines under `src/`, 34,897
when files named `tests.rs` and `test_formats.rs` are excluded) with emphasis
on changes landed after those audits:

- explicit `RunState` and `FinalizedRun` ownership;
- `RefCell` to `RwLock` and `Rc` to `Arc` migrations;
- optional Rayon bibliography rendering and its measured crossover;
- the unified cached `ReferenceSorter` and multilingual sort-key support;
- cached interactive-session inputs and the document bibliography facade;
- all open children of the existing engine-review epic `csl26-8m2p`.

The review combined line-by-line inspection of the recent concurrency,
bibliography, sorting, setup, and disambiguation changes with crate-wide scans
for `unsafe`, panic macros, `unwrap`/`expect`, indexing exceptions, allocation
sites, locks, I/O, parsing, and large modules. The Rust review-smell script was
also run against the changed Rust files. Findings from the earlier audits were
checked against current code and beans before new beans were created.

## Architecture Overview

The crate has four principal layers:

1. `api/` resolves styles and references and exposes batch
   (`format_document`) and interactive (`DocumentSession`) workflows. It also
   collects structured warnings and forward-compatibility diagnostics.
2. `processor/` owns the processing pipeline. `Processor` is now immutable
   style + bibliography + locale context; `RunState` accumulates ordered
   citation facts, and `FinalizedRun` gates bibliography rendering. Citation,
   bibliography, note, document, disambiguation, and grouped-rendering modules
   implement focused stages behind that facade.
3. `values/` resolves typed style components—contributors, dates, titles,
   locators, numbers, terms, and variables—into intermediate render values.
4. `render/` implements the `OutputFormat` abstraction and converts processed
   templates into plain text, HTML, Djot, Markdown, LaTeX, Typst, Org, and rich
   intermediate forms. `sorting.rs`, `sort_support.rs`, and
   `sort_partitioning.rs` provide cached, locale-aware ordering shared across
   bibliography and grouping workflows.

The primary batch flow is:

`StyleInput`/`RefsInput` resolution → warning scan → immutable `Processor` →
`begin_run` → ordered citation/nocite registration and rendering → `finalize`
→ cited bibliography selection/rendering → output-format conversion.

From a performance perspective, expensive style resolution, reference parsing,
disambiguation hints, collators, and sort keys are mostly moved out of
pairwise or per-component loops. From a correctness perspective, typestate now
prevents bibliography rendering before registration is declared complete.
From a maintenance perspective, the main residual weakness is orchestration
duplication at the API/document boundary, already tracked by `csl26-aawl`.

## Idiomatic Rust and Safety

### Strong practices

- `RunState::finalize(self) -> FinalizedRun` encodes the register-before-render
  contract in ownership and types instead of relying on call-order comments.
- `Processor` is reusable and `Send + Sync`; per-run mutation is no longer
  hidden in the facade. Borrowed references and lifetime parameters are used
  through sorting and rendering without self-referential storage.
- `Result` and `thiserror` define public error boundaries. Recoverable input,
  parsing, missing-reference, and compound-set failures are propagated rather
  than panicking.
- `Cow`, `Arc`, traits, and generics are used at genuine boundaries:
  configuration is borrowed unless a merge is required, shared render config
  avoids deep cloning, and `OutputFormat` keeps format dispatch statically
  typed in the processing core.
- Poisoned `RwLock`s are handled explicitly rather than with `unwrap`. This is
  a deliberate availability choice; the maps contain monotonic render state.
- Strict workspace lints reject production `unwrap`, `expect`, panic macros,
  unchecked indexing, unjustified allows, and undocumented public items.

### Safety audit

No new or unjustified production `unsafe` was found. All executable unsafe
operations are in `src/ffi/mod.rs`, which has a module-level
`allow(unsafe_code, reason = "FFI interface")`. Public C functions contain
`# Safety` contracts; pointer conversion is centralized; null inputs,
invalid UTF-8, and allocation ownership are tested. Test-only unsafe calls
exercise those C boundaries. No safe alternative exists for the raw-pointer C
ABI itself, and no unsafe code is used to optimize engine hot paths.

The remaining production `expect`/`unwrap` exceptions are narrow and carry
lint reasons: ICU compiled-data bootstrap, length-checked access, or
scanner-guaranteed branches. No newly panic-prone indexing path was found.

### Less idiomatic or fragile areas

- `Processor` and `Renderer` still expose more fields than callers need.
  External construction can bypass invariants; this remains appropriate for
  the existing low-severity cleanup bean `csl26-dr0r`, not a duplicate bean.
- `sort_citation_items` returns a plain `Vec`, so it cannot report a missing
  reference at the sorting boundary. The audit fix preserves unresolved items
  for the renderer's existing `Result` path rather than broadening this public
  signature.
- Recovering a poisoned render-state lock favors continued output over an
  explicit error. That is defensible for assign-once maps, but the invariant
  should remain documented if those map values become compound state.

## Performance Analysis

### Improvements confirmed in current code

- `ReferenceSorter` compiles the sort template and caches derived author,
  title, type, date, and field values before stable sorting. Collation and
  leading-article stripping are no longer repeated for both operands of every
  comparison.
- `EntryRenderContext` merges and `Arc`-wraps shared configuration once per
  bibliography pass. Each entry clones only `Arc` handles.
- `DocumentSession` caches resolved style/reference inputs and constructs a
  fresh lightweight run rather than reparsing and cloning the complete model
  for every mutation.
- The parallel bibliography work was benchmarked rather than assumed useful.
  Measurements in `PARALLEL_BIBLIOGRAPHY_RENDERING.md` show it is effectively
  neutral at 10–400 entries after config hoisting, so Rayon remains opt-in.

### Remaining allocation and hot-path findings

#### P1. Document bibliography output renders the same subset twice (Medium)

`render_document_bibliography` first builds `content` through
`render_grouped_bibliography_inner`, then independently builds `entries`
through `process_selected_references_with_format` or `process_references_with_format`
(`processor/bibliography/grouping.rs:585-615`). In the normal flat document
path, the content pass renders **all loaded references** and only afterward
clones/filter-selects cited entries in `render_with_legacy_grouping`
(`grouping.rs:487-516, 676-684`). The cited subset is rendered again for
`entries`.

For a document citing `c` items from a library of `n`, this performs roughly
`O(n + c)` entry rendering when `O(c)` should be possible, plus duplicate sort,
template, metadata, and allocation work. The cost is especially visible for
large personal libraries and short documents. Custom-group and partition
branches also produce content and flat entry data separately, which creates a
future drift risk around group-local templates and disambiguation.

Recommendation: make the cited, ordered `Vec<ProcEntry>` the shared product and
derive both string content and public entry data from it. Preserve group-local
semantics and benchmark a large-library/small-cited-set case. Tracked as
`csl26-plaz`.

#### P2. Group-local processing clones and repeatedly sorts reference models (Medium/Low)

`build_group_local_hints` clones every selected `Reference` into a temporary
`Bibliography` before running disambiguation (`grouping.rs:151-184`).
`entries_for_bibliography_group` also calls `sorted_id_stubs`, which sorts the
entire bibliography for each independently rendered group/block
(`grouping.rs:697-735`). For `g` blocks over `n` references this can approach
`g × O(n log n)` sorting plus deep reference clones.

Trade-off: the temporary owned bibliography gives `Disambiguator` a simple,
safe lifetime boundary, so changing it to accept a borrowed subset adds API
complexity. Address this only with the document-bibliography benchmark in
`csl26-plaz`; cache the sorted ID spine per facade call first, then consider a
borrowed disambiguation view if profiling still identifies cloning.

#### P3. Small avoidable formatting allocations remain (Low)

Examples include repeated integer `to_string` calls while padding a year
(`values/date.rs:93-110`) and temporary formatted delimiters in citation-list
joining (`processor/citation.rs:48-63`). These are real hot-path allocations,
but individually small. Broad replacement without benchmarks would make the
code less readable for uncertain gain. Treat them as candidates for a measured
rendering benchmark wave, not as mechanical clone-count cleanup.

No blocking I/O or repeated input parsing was found in the core render loops.
Filesystem reads and serde parsing stay at `StyleInput`/`RefsInput` and FFI
construction boundaries.

## Bug and Correctness Review

#### C1. Sorted citations silently discarded missing references (High, fixed)

Before this audit, `sort_citation_items` used `filter_map` while pairing items
with references (`processor/setup.rs:543-550`). With any `citation.sort`
configured, a citation containing one valid ID and one missing ID lost the
missing item before rendering. `process_citation` could therefore return a
successful partial citation despite documenting `ReferenceNotFound`.

The fix retains `(CitationItem, Option<&Reference>)`, sorts resolved items
before unresolved items, and leaves unresolved items in the result so the
existing renderer error path executes. The regression test
`citation_sort_preserves_missing_reference_error` is independent of generated
output and fails on the old behavior. Tracked and completed as `csl26-pcaq`.

#### C2. Year-suffix order ignores multilingual/locale sort policy (Medium)

Final bibliography sorting constructs
`ReferenceSorter::with_bibliography_config`, which applies
`sorting.multilingual`, `sort-as`, preferred transliterations, and an explicit
sorting locale (`processor/setup.rs:506-526`). Disambiguation receives the same
group sort shape but constructs `ReferenceSorter::new`
(`processor/disambiguation.rs:701-728`), which uses uniform original-text keys
and the active render locale.

Scenario: two same-author, same-year Cyrillic works have `sort-as` title keys
whose order differs from their original titles. A romanized bibliography can
display work B before work A while assigning B the `b` suffix and A the `a`
suffix. The visible bibliography and its disambiguation letters disagree.

Recommendation: thread the effective bibliography sort-key configuration into
`Disambiguator`, including group-local construction, and add a fixture whose
original and hidden-key orders deliberately oppose each other. Tracked as
`csl26-2tbe`.

#### C3. Compound-set forgiving constructor remains silent (Low, inherited)

The forgiving constructor still replaces invalid compound sets with an empty
map while the `try_` constructor returns an error. This is already covered by
the silent-fallback batch `csl26-ol1j`; no duplicate was created.

#### C4. Document parser syntax/warning differences remain (Low/Medium, inherited)

Djot and Markdown deliberately support different syntax, but malformed or
unsupported constructs can still be silently retained as source text. This is
already tracked by `csl26-esq8`; no duplicate was created.

## Maintainability and Design

The recent state refactor is the strongest maintainability improvement:
`Processor` is again a description of immutable processing context, and run
ordering is visible in signatures. Moving sorting to a single cached stack
also removes two previously divergent implementations.

The most fragile remaining boundaries are:

- `api/document.rs` and `api/session.rs`: preparation, warning collection,
  format dispatch, and result shaping remain substantially duplicated. Use
  existing bean `csl26-aawl`.
- `processor/bibliography/{mod,grouping}.rs`: selection, grouping, rendering,
  substitution, metadata, document shaping, and parallel dispatch cross the
  same two modules. The two-pass finding is a symptom of this responsibility
  density; `csl26-plaz` should establish one internal rendered-entry product.
- `processor/disambiguation.rs`: the cascade is necessarily complex, but sort
  policy is supplied indirectly as a `GroupSort` without the configuration
  required to interpret multilingual keys. `csl26-2tbe` should make that
  dependency explicit.
- `processor/rendering/grouped/core.rs` and the document parsers contain
  scanner/format state with justified indexing exceptions. Their invariants
  are currently well commented and tested; edits should remain narrow and
  preserve exact-output tests.

Existing open low-level cleanup (`csl26-dr0r`), punctuation-format awareness
(`csl26-ztxq`), number-form behavior (`csl26-c361`), and silent warnings
(`csl26-ol1j`) remain valid. None were duplicated.
