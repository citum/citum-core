Integration

Developer

Citum exposes several integration surfaces depending on your runtime. Pick the one that fits your environment — or start from one of the recipes below.

Install

Prebuilt binaries for Linux, macOS (arm64), and Windows (via Git Bash). The installer verifies SHA-256 checksums and installs to ~/.local/bin.

Install (prebuilt binary)
curl -fsSL https://github.com/citum/citum-core/releases/latest/download/install.sh | sh

Select components with --components (citum, citum-server, citum-migrate, or all). CITUM_COMPONENTS remains available for automation. From source: cargo install citum --locked (also: citum-server, citum-migrate). JavaScript/TypeScript (WASM): npx jsr add @citum/engine. See the CLI section for component examples and common commands.

Start here if you're building…

…and you just want to format a document

Render a Markdown or Djot manuscript with citations into HTML, LaTeX, Typst, or PDF using the citum CLI.

…a static site or docs build

Wire citum into a Makefile, npm script, or CI job to render citations as part of your existing build.

…an editor or plugin

Run citum-server as a long-lived JSON-RPC process and call it from your editor, IDE plugin, or word-processor add-in.

…and you're already writing Rust

Depend on citum-engine as an ordinary crate and call its API directly — no CLI, WASM, or RPC boundary. The path for static-site generators, mdbook preprocessors, and custom publishing toolchains written in Rust.

…a Deno, Node, or browser app

Import @citum/engine from JSR and call renderBibliography / renderCitation directly. TypeScript types included; the WASM is cached after first load.

…a house style for a publisher or journal

Express your style as YAML that extends a pinned parent (APA, Chicago, …) instead of forking and re-maintaining XML.

Integration surfaces

Reference cards for each integration path. Each links to deeper documentation below.

CLI

Render references, process Djot or Markdown documents, validate styles, and inspect schemas from the command line. The simplest integration path for build pipelines and scripted workflows.

Rust library

Depend on citum-engine as a crate and call its API directly. The native path for Rust binaries and libraries — no process spawn, no WASM, no RPC.

JSR (Deno / Node / browser)

Import the engine from the JSR registry as @citum/engine. The simplest path for TypeScript / JavaScript projects on Deno, Node, or the browser — types included.

Typst PDF

Render citations with Citum and compile to PDF using embedded Typst Rust crates. No TeX, no FFI step — a pure Rust pipeline.

C FFI

Call the engine from any language with a C FFI. Current bindings cover Lua (for LuaLaTeX). The shared library exports a stable C-compatible surface.

JSON-RPC server

Run Citum as a long-lived process and call it over stdin/stdout or HTTP. Suitable for editor integrations, language-server patterns, and applications that need live formatting without process-spawn overhead.

Interactive HTML

Drop one stylesheet and one script into your rendered page and citations become clickable, scroll to their bibliography entry, and show hover previews. No build step.

CLI

Pre-built binaries for Linux, macOS, and Windows — no Rust toolchain required. Or build from source with Cargo.

Install (pre-built binary)
curl -fsSL https://github.com/citum/citum-core/releases/latest/download/install.sh | sh

Installs citum by default. To pick a different subset, pass --components with a comma-separated list of citum, citum-server, citum-migrate, or all. CITUM_COMPONENTS remains available for automation: 

# everything
curl -fsSL https://github.com/citum/citum-core/releases/latest/download/install.sh | sh -s -- --components all

# citum + citum-migrate, no server
curl -fsSL https://github.com/citum/citum-core/releases/latest/download/install.sh | sh -s -- --components citum,citum-migrate
Install (from source)
cargo install citum
Common commands
# Render a bibliography (plain text)
citum render refs -b references.json -s styles/apa-7th.yaml

# Render a bibliography (HTML)
citum render refs -b references.json -s styles/apa-7th.yaml -f html

# Process a Djot document with citations
citum render doc paper.djot -b references.json -s styles/apa-7th.yaml

# Validate a style
citum style validate styles/my-style.yaml

# Inspect the style schema
citum schema --out-dir ./schemas

See Write citations in Djot for a full end-to-end example, or the Typst and LuaLaTeX recipes for PDF publishing workflows. For CBOR binary performance, pass -b data.cbor -s style.cbor — compile YAML to CBOR first with citum convert style styles/apa-7th.yaml -o apa-7th.cbor.

Rust library

The engine is published as the citum-engine crate. Depend on it from any Rust binary or library and call its API directly — the natural path for static-site generators, mdbook preprocessors, custom publishing toolchains, and any tool that already runs in the Rust ecosystem.

Add the dependency
cargo add citum-engine
Format a document
use citum_engine::{
    Bibliography, CitationOccurrence, CitationOccurrenceItem, FormatDocumentRequest,
    OutputFormatKind, StyleInput, format_document,
};

let refs: Bibliography = serde_json::from_str(&std::fs::read_to_string("refs.json")?)?;
let result = format_document(FormatDocumentRequest {
    style: StyleInput::Path("apa-7th.yaml".to_string()),
    locale: Some("en-US".to_string()),
    output_format: OutputFormatKind::Html,
    refs,
    citations: vec![/* CitationOccurrence per citation in document order */],
    document_options: None,
})?;

See Embed Citum in a Rust binary or library for the full recipe — programmatic construction, loading styles and references from disk, document formatting, and how to wire this into your own document model (mdbook, static-site generators, custom tools).

JSR (Deno / Node / browser)

The engine is published to the JSR registry as @citum/engine. It wraps the WASM bridge below, packaged for direct import from Deno, Node, or the browser with TypeScript types included. Best fit for Deno and Node, where the ~5 MB WASM is downloaded once into the lockfile or node_modules and loaded from disk thereafter. In the browser, ship through a bundler and serve with Brotli or gzip at your CDN — the compressed payload is closer to 800 KB; JSR itself serves the WASM uncompressed. The raw WASM bridge is the lower-level escape hatch.

Deno
import init, { renderBibliography, renderCitation } from "jsr:@citum/engine";
await init();
Node 20+ (ESM)
# In your project
npx jsr add @citum/engine
import init, { renderBibliography, renderCitation } from "@citum/engine";
import { readFile } from "node:fs/promises";
import { fileURLToPath } from "node:url";

// Node's fetch can't load file:// URLs, so feed init() the wasm bytes
const jsUrl = import.meta.resolve("@citum/engine");
const wasmUrl = jsUrl.replace("citum_bindings.js", "citum_bindings_bg.wasm");
await init({ module_or_path: await readFile(fileURLToPath(wasmUrl)) });
Render (any runtime)
const style = "id: apa-7th";
const refs = JSON.stringify([
  {
    id: "darcus2026",
    class: "monograph",
    type: "book",
    title: "Citum: A Modern Citation Engine",
    author: [{ family: "D'Arcus", given: "Bruce" }],
    issued: "2026",
  },
]);

const html = renderBibliography(style, refs);

In a browser, use the same call sequence with a bundler that handles wasm-pack --target web output (Vite, esbuild, Webpack 5+, Rollup with @rollup/plugin-wasm): the bundler resolves the wasm asset for init() automatically, so the snippet looks like the Deno one. The package also exports formatDocument, validateStyle, materializeStyle, and getStyleMetadata. See the Interactive HTML section below to make the rendered output clickable in a page.

WASM

The lower-level WASM bridge is built from crates/citum-bindings using wasm-pack. The JSR package above wraps this bridge for Deno / Node / browser; build it yourself only when you need a custom target or to embed the engine inside another bundle. The bridge is the integration layer used by Citum Hub.

Build
cd crates/citum-bindings
wasm-pack build --target nodejs
Usage (Node.js)
import { render_bibliography } from './pkg/citum_bindings.js';

const result = render_bibliography({
  style: styleYamlString,
  references: referencesJson,
  locale: 'en-US',
});

The WASM module requires rebuilding when the engine changes. TypeScript types are generated by wasm-pack alongside the module.

C FFI

The crates/citum-bindings crate exposes a C-compatible FFI surface. Build the shared library first, then link against it from your target language.

Build shared library
cd crates/citum-bindings
cargo build --release
# Output: target/release/libcitum_bindings.{so,dylib,dll}

Lua / LuaLaTeX

Experimental bindings (citum-labs) let a LuaLaTeX document call the Citum engine for citations and bibliography rendering via LuaJIT FFI or JSON-RPC. Two-pass compilation; no Biber or external bibliography tool required. See the LuaLaTeX integration recipe for package usage and build instructions.

JSON-RPC server

The citum-server crate provides a long-running process that accepts JSON-RPC requests over stdin/stdout or HTTP. Use it for editor integrations, plugins, or web applications that need live citation formatting without process-spawn overhead per request.

Install and start
cargo install citum-server
citum-server                          # stdin/stdout JSON-RPC
citum-server --http --port 8765       # HTTP JSON-RPC

Methods: format_document, render_citation, render_bibliography, validate_style. See the editor / plugin recipe for a working request, response, and Node client. The HTTP server is included in the default citum-server build through the citum-server/http feature (axum); use cargo install citum-server --no-default-features only for a stdio-only binary. For the full schema, see Reference → JSON Schemas or browse docs/schemas/ directly.

Interactive HTML

Citum's HTML output is annotated with stable hooks (.citum-citation, data-ref, .citum-bib-entry). Including citum-interactive.{css,js} turns those annotations into a reading-aware experience: click a citation to jump to its bibliography entry, hover for a preview, and toggle between inline-numbered and footnote presentations. No build step, no framework, no JS API to wire up — just drop the two assets into your page.

Drop-in include
<link rel="stylesheet" href="https://docs.citum.org/assets/citum-interactive.css">
<script src="https://docs.citum.org/assets/citum-interactive.js"></script>

See the live demo for a runnable page. The assets are MPL-2.0 and free to vendor into your own site.

Plugins for word processors

Citum doesn't currently ship a Word, LibreOffice, Pages, or Google Docs plugin — but the integration shape is well-suited to it. The JSON-RPC server is the natural fit for desktop add-ins (one long-lived process per editor session); the WASM bridge is the natural fit for browser-based add-ons. If you're interested in building one, open a thread on GitHub Discussions — we're happy to support the work.
